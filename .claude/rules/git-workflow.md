# Git 工作流规则

> 来源：ADR-006

## 核心理念

本地提交要勤（方便回溯），推送看时机（完整才推）。

## 行为表

| 场景 | 行为 |
|------|------|
| 任何代码修改完成 | 立即 `git commit`，保持提交粒度细 |
| 阶段性完成（bug 修完、功能做完） | commit → 自动 `git push origin main` |
| 快速迭代中（多个小提交短时间） | 仅 commit 不 push，阶段性完成再一起推 |
| 会话结束时仍有未推送提交 | 提醒用户，询问是否推送 |
| 用户明确说"提交"/"推送" | commit + push |
| 推送失败（网络/权限） | 告知用户，提交保留本地，不阻塞，不重试 |

## 自动判断 push 的时机

1. 完整功能/修复完成 → push
2. 用户说"提交"/"推送" → push
3. 连续修改多个相关文件，最后一个完成 → push
4. 仅改错字或小调整，后续还有改动 → 只 commit 不 push
5. 推送失败 → 不重试，后续 commit 继续累积
