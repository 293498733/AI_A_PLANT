# Architecture: 全流程自动化工程系统

## 核心抽象

整个系统围绕一个 **状态自动机** 构建，它把任何工程任务映射到统一的流水线中：

```
需求(RAW) → 解析(Structured) → 计划(Executable) → 执行(Code) → 验证(Feedback) → 完成(Done)
                              ↕                      ↕
                         自适应上下文          人类确认点
```

## 状态自动机定义

### 阶段: 0-UNKNOWN → 1-PARSED → 2-PLANNED → 3-EXECUTING → 4-VERIFYING → 5-DONE

每条边有条件和动作：
```
0→1: 输入非空 → 解析需求
1→2: 需求结构化 → 生成计划
2→3: 计划已确认 → 执行代码
3→4: 代码已产出 → 运行验证
4→5: 验证通过 → 标记完成
    验证失败 → 回退到2或3
```

### 自适应模式（从0开始 vs 半路接手）

| 维度 | 从0开始 | 半路接手 |
|------|---------|---------|
| 入口 | 需求文档/描述 | `接管协议` |
| 第一步 | 创建脚手架 | 分析现有仓库 |
| 上下文来源 | 设计文档 | 代码库+学习记录 |
| 风险 | 设计错误 | 破坏现有功能 |
| 验证策略 | 测试驱动 | 回归测试+diff检查 |

## 接管协议 (Handover Protocol)

当面对一个已有工程时，系统自动执行：

```
PHASE 1: 仓库侦察 (Repo Reconnaissance)
  1. 目录树分析（文件结构、语言分布）
  2. 构建系统检测（package.json / Cargo.toml / CMakeLists.txt ...）
  3. 测试框架检测
  4. 文档扫描（README、CONTRIBUTING、CLAUDE.md、DESIGN.md）
  5. Git历史分析（活跃分支、最近变更、提交模式）
  6. 依赖分析
  7. CI/CD配置检测

PHASE 2: 心智模型重建 (Mental Model Reconstruction)
  1. 提取领域实体（类、函数、组件、路由）
  2. 构建数据流图
  3. 理解设计约束
  4. 识别技术债务

PHASE 3: 介入点推荐 (Entry Point Recommendation)
  1. 分析当前开发状态（原型？稳定版？遗留系统？）
  2. 评估"改动风险等级"
  3. 推荐最小侵入式介入策略
  4. 生成接管报告
```

## 核心引擎设计

### 1. Requirement Engine (需求引擎)
```
输入: 自然语言需求 / PRD / Issue / 用户提示
处理: 
  - 结构化解构（目标、范围、约束、验收标准）
  - 模糊性检测+追问
  - 已有上下文对齐
输出: 结构化需求文档
```

### 2. Plan Generator (计划生成器)
```
输入: 结构化需求 + 上下文
处理:
  - CEO/工程/设计 三重评审（调用对应skill）
  - 任务分解（文件粒度）
  - 依赖排序
  - 风险评估
输出: 可执行任务列表
```

### 3. Code Executor (代码执行器)
```
输入: 任务列表
处理:
  - 按依赖顺序执行
  - 每次变更前读取文件
  - 每次变更后验证语法/编译
  - 自动提交（Continuous Checkpoint）
输出: 变更后的代码库
```

### 4. Verification Loop (验证闭环)
```
输入: 代码变更
处理:
  - 编译/语法检查
  - 测试运行
  - 质量检查
  - 回归验证
  - 人类确认点（可选）
输出: 通过/失败反馈
```

## 技能集成层

所有gstack skill作为插件集成：

| Skill | 阶段 | 用途 |
|-------|------|------|
| plan-ceo-review | 计划阶段 | 审查目标是否足够大 |
| plan-eng-review | 计划阶段 | 审查架构合理性 |
| plan-design-review | 计划阶段 | 审查UI/UX设计 |
| investigate | 执行阶段 | 调试错误 |
| qa | 验证阶段 | 端到端测试 |
| review | 验证阶段 | diff评审 |
| ship | 完成阶段 | 提交PR/部署 |
| health | 全阶段 | 代码质量监测 |
| learn | 全阶段 | 持久化学习记录 |
| context-save/restore | 全阶段 | 状态持久化 |

## 自举 (Bootstrap)

系统用自己构建自己：
1. 第一阶段：手动构建核心自动机
2. 第二阶段：用核心自动机构建技能集成
3. 第三阶段：系统自举后用于自身迭代
