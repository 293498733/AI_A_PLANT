# 接管协议实战指南

> 如何在"半路接手"场景中正确使用接管协议

## 场景一：接手快速原型

**仓库状态：** PROTOTYPE（<10文件，无测试，无CI）
**目标：** 完善功能，添加基础设施

```
动作：
1. 侦察完成 → 状态是PROTOTYPE
2. 理解：代码少但可能混乱
3. 建议：先加测试框架 → 再重构核心逻辑 → 再添加功能
```

**典型流程：**
```
1. 创建测试配置文件
2. 为核心函数写单元测试
3. 边重构边确保测试通过
4. 添加CI配置
5. 编写README
```

**风险：LOW — 代码量小，重构风险低**

## 场景二：接手活跃开发项目

**仓库状态：** ACTIVE_DEV（有测试，有CI，多人协作）
**目标：** 添加新功能

```
动作：
1. 侦察完成 → 状态是ACTIVE_DEV
2. 理解：代码有组织，但有其他人正在工作
3. 建议：遵循已有模式，最小范围修改
```

**典型流程：**
```
1. 阅读ARCHITECTURE.md了解设计
2. 阅读已有功能代码了解模式
3. 按已有模式添加新功能
4. 运行完整测试套件
5. 仅添加必要的测试
```

**风险：MEDIUM — 有回归风险，但安全网存在**
**关键原则：最小化对现有文件的修改**

## 场景三：接手稳定生产项目

**仓库状态：** STABLE（有测试，有CI，有CHANGELOG，多人使用）
**目标：** 修复bug或添加小功能

```
动作：
1. 侦察完成 → 状态是STABLE
2. 理解：代码成熟，有完整安全网
3. 建议：非常小心，每个变更都需要充分验证
```

**典型流程：**
```
1. 阅读CHANGELOG了解最新变更
2. 创建新分支
3. 运行完整测试确认基线
4. 最小化变更（不超过3个文件）
5. 完整测试+回归测试
6. 创建PR请求审核
```

**风险：HIGH — 直接影响用户**
**关键原则：绝不跳过测试**

## 场景四：接手遗留系统

**仓库状态：** LEGACY（大量文件，无测试，无CI）
**目标：** 重构或迁移

```
动作：
1. 侦察完成 → 状态是LEGACY
2. 理解：代码量大，缺乏安全网
3. 建议：先写测试安全网，再小步重构
```

**典型流程：**
```
1. 定义接口边界
2. 为核心路径写集成测试
3. 逐模块添加单元测试
4. 每次重构不超过1个模块
5. 每个模块重构后完整测试
6. 添加CI
```

**风险：CRITICAL — 没有安全网，修改可能导致未知问题**
**关键原则：先写测试，再改代码。绝不跳过测试阶段。**

## 通用规则

### 每次都做的事
1. ✅ 运行 `tree` 查看目录结构
2. ✅ 读取 README.md
3. ✅ 读取 CLAUDE.md（如果有）
4. ✅ 检查 git log
5. ✅ 检测构建系统和测试框架

### 绝对不要做的事
1. ❌ **不要不读文件就修改** — 每次修改前先 read
2. ❌ **不要跳过测试** — 哪怕"只是重构"
3. ❌ **不要一次改太多文件** — 每次不超过3个
4. ❌ **不要忽略CLAUDE.md** — 那是前人的血泪教训

### 黄金法则

> **在完全理解之前，不要修改任何文件。**
> 
> "理解"的标准：
> - 你知道每个文件是做什么的
> - 你知道数据流是怎样的
> - 你知道测试覆盖了哪些路径
> - 你知道风险在哪里
