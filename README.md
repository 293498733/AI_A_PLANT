# 全流程自动化工程系统

> **从原始需求 → 计划 → 代码 → 验证 → 交付 的全自动闭环**

## 一句话

这是一个 **Meta-Orchestrator** —— 把"需求→计划→代码→验证"这个工程师循环本身自动化。任何AI agent加载本系统后，就获得了从零构建或接手任意代码库的完整工作流。

## 核心能力

| 能力 | 说明 |
|------|------|
| 🔄 **全自动闭环** | 需求输入 → 计划生成 → 代码执行 → 验证交付，零人工介入 |
| 🧩 **自适应模式** | 自动判断"从0开始"还是"半路接手"，选择最佳策略 |
| 🔌 **Skill集成** | 调用 gstack 所有 skill（CEO评审/工程评审/QA/安全审计...） |
| 🛡️ **接管协议** | 接手已有工程的标准化流程：先侦察→再理解→最后动手 |
| 🔙 **自动回滚** | 错误检测 → 根因分析 → 自动修复或回滚 |
| 📍 **连续检查点** | 每个原子变更自动git提交，随时可回溯 |
| 🧠 **自适应策略** | 根据执行历史动态调整：连续成功加速，连续失败保守 |

## 快速开始

### 方式一：从零开始构建

直接告诉系统你的想法：

```bash
# 任何AI agent加载本系统后
"帮我构建一个CLI工具，用于..."
```

系统自动：
1. 检测到空目录 → **GREENFIELD** 模式
2. 解析需求 → 生成计划 → 评审
3. 创建项目脚手架 → 实现功能
4. 测试 → 验证 → 输出

### 方式二：接手已有工程

```bash
# 在已有代码库中
"帮我添加用户认证功能"
```

系统自动：
1. 检测到已有仓库 → **BROWNFIELD** 模式
2. 执行接管协议（侦察→理解→建议）
3. 生成最小侵入计划
4. 执行 → 回归验证 → 交付

### 方式三：半路接手

```bash
"接手这个项目，优化性能"
```

接管协议自动分析：
- 📊 语言/构建/测试/CI
- 🧠 实体/数据流/约束/技术债务
- 💡 最佳介入点推荐

## 核心概念

### 状态自动机

```
UNKNOWN → PARSED → PLANNED → EXECUTING → VERIFYING → DONE
                    ↑           │            │
                    └───────────┴────────────┘
                         回滚/修复回路
```

### 自适应模式

| 模式 | 场景 | 策略 |
|------|------|------|
| **GREENFIELD** | 空目录/新项目 | 探索优先，全面构建，测试驱动 |
| **BROWNFIELD** | 已有仓库 | 安全优先，最小侵入，回归验证 |

### 接管协议 (Handover Protocol)

面对已有工程时的标准化流程：

```
PHASE 1: 仓库侦察
  - 语言/构建/测试/CI检测
  - 目录结构分析
  - Git历史分析
  - 仓库状态评估

PHASE 2: 心智模型重建
  - 领域实体提取
  - 数据流推断
  - 设计约束提取
  - 技术债务识别

PHASE 3: 介入点推荐
  - 风险评估
  - 最小侵入策略
  - 首次行动推荐
```

## 设计哲学

### 1. 状态驱动，而非流程驱动

系统由状态自动机驱动，每个阶段只关心"当前状态+事件→下一状态"，而不是"步骤1→2→3"的流水线。这让系统能够自然处理错误、回滚和分支。

### 2. 接管优先，修改其次

面对已有代码库，第一步永远是**理解**而不是修改。接管协议的"侦察→理解→建议"三阶段确保系统在完全理解前不会破坏任何东西。

### 3. 验证闭环，而非测试通过

验证不仅是"测试通过了"，而是多维度检查：编译、类型、lint、测试、安全、diff评审、QA。每个维度都可以独立开关。

### 4. Skill即插件

每个gstack skill都是一个可插拔的能力单元。系统不重复实现这些能力，而是在正确的时机路由到正确的skill。

### 5. 自适应，而非固定策略

系统根据执行历史动态调整策略：连续成功就加速（减少确认点），连续失败就保守（增加验证），从错误中学习。
