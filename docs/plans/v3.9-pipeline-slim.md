# v3.9 管线瘦身计划

> 来源：2026-05-15 进销存 WMS 分析后续 — Phase 5 不应是 AI 阶段 + CI 模式不可用于真实 CI
> 状态：方案定型，待排期

---

## 背景

v3.8 修了 6 个"失控"问题（环境误判、状态二值化、失败传播），但两个更深层的设计问题仍未解决：

### 问题 1：Phase 5 是 AI 阶段，不是验证阶段

当前 `05-build.yaml` 的 prompt 包含"尝试修复编译错误"，这让 goose 在无任务上下文的情况下猜测性改代码。60 turns 中实际编译只需 30 秒，其余时间烧在"读错误 → 改代码 → 重试"上。

Phase 5 的正确定位是**确定性验证**，不是 AI 编码。编译和测试应该直接跑命令，结果写入报告文件。代码修正应该是 Phase 4 的职责，Phase 5 只负责判定通过/失败。

### 问题 2：CI 模式是"静默跳过"，不是"真实 CI"

`error_handler.py` 的 CI 逻辑对所有失败统一 `SKIP`。环境问题（JAVA_HOME 缺失）和业务问题（代码 bug）被等同对待。CI 模式下 pipeline 永远返回 exit 0，无论实际完成度。

真实 CI 需要：环境类失败 → 立即终止非零退出；业务类失败 → skip-continue 但最终非零退出。需要区分失败类型。

### D7 — Phase 5 脚本化

**现状**：`05-build.yaml` 是 goose recipe，AI 尝试编译→修复错误→测试→输出报告。

**方案**：去掉 goose，改为 `pipeline/runner.py` 直接调用的确定性函数。

`pipeline.yaml` 变更：
```yaml
- id: phase5
  name: "编译与测试"
  run_direct: true          # 新字段：非 goose 阶段
  prerequisite: task_graph
  state_value: build_done
```

`pipeline/runner.py` 新增 `_run_phase5_direct()`：
```python
def _run_phase5_direct(P: Path, AD: Path, OUT: Path, profile: dict, logger) -> tuple[bool, str]:
    """确定性执行编译+测试，不经过 goose。"""
    commands = profile.get("commands", {})
    compile_cmds = commands.get("compileOnly") or commands.get("build") or []
    test_cmds = commands.get("test") or []

    result_lines = ["# 构建验证结果\n"]
    all_passed = True

    # 编译
    result_lines.append("## 编译\n")
    for cmd in compile_cmds:
        proc = subprocess.run(cmd, cwd=str(P), shell=True, capture_output=True, text=True, timeout=600)
        status = "通过" if proc.returncode == 0 else "失败"
        all_passed = all_passed and (proc.returncode == 0)
        result_lines.append(f"- 状态：{status}")
        if proc.returncode != 0:
            result_lines.append(f"- 命令：`{cmd}`")
            for line in proc.stderr.splitlines()[-10:]:
                result_lines.append(f"  - {line}")

    # 测试
    if test_cmds:
        result_lines.append("\n## 测试\n")
        for cmd in test_cmds:
            proc = subprocess.run(cmd, cwd=str(P), shell=True, capture_output=True, text=True, timeout=600)
            status = "通过" if proc.returncode == 0 else "失败"
            all_passed = all_passed and (proc.returncode == 0)
            result_lines.append(f"- 状态：{status}")

    result_lines.append(f"\n## 总结\n- 构建{'通过' if all_passed else '未通过'}")

    output_path = OUT / "05-build.md"
    output_path.write_text("\n".join(result_lines), encoding="utf-8")
    return all_passed, str(output_path)
```

### D8 — CI 退出码区分

**现状**：`handle_error()` 和 `handle_task_error()` 在 CI 模式下统一 SKIP。pipeline.main() 始终正常返回（exit 0）。

**方案**：新增 `CIFailure` 分类 + `pipeline/runner.py` 收集跳过/失败计数，最终根据严重程度决定退出码。

`error_handler.py` 变更：
```python
class CIFailure(Enum):
    ENV = "env"       # 环境类失败 → 应终止
    BUSINESS = "business"  # 业务类失败 → skip-continue

def classify_failure(error_msg: str) -> CIFailure:
    """根据错误信息分类失败类型。"""
    env_keywords = [
        "JAVA_HOME", "not defined correctly", "not found",
        "command not found", "is not recognized",
        "Permission denied", "cannot find",
    ]
    error_lower = error_msg.lower()
    for kw in env_keywords:
        if kw.lower() in error_lower:
            return CIFailure.ENV
    return CIFailure.BUSINESS
```

`pipeline/runner.py` 退出码：
```python
# main() 末尾
if args.ci:
    if env_failures > 0:
        sys.exit(2)   # 环境问题 → 需要人工修复
    elif skipped_phases > 0:
        sys.exit(1)   # 部分跳过 → 需检查
    else:
        sys.exit(0)   # 完全通过
```
