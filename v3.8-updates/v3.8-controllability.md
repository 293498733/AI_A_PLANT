## 背景

2026-05-14 在 `D:\MyPrj\new_进销存_ai_CODE`（进销存 WMS — 返修通知单核对明细按规格型号汇总）运行全流程管线：

| 阶段 | 结果 | 问题 |
|------|------|------|
| Phase 0-3.5 初始化→拆分 | ✅ 全部通过 | — |
| Phase 4 任务图执行 | ⚠️ 2 失败 + 3 阻塞 | JAVA_HOME 导致 mvn compile 验证失败 |
| Phase 5 编译与测试 | ❌ Goose 卡死 | 同因，4m36s 无输出 |
| Phase 6-7 审查→交付 | ✅ 形式通过 | 但审的是不完整代码 |

事后确认：AI 实际已成功写入全部代码变更（后端 40 行 + 前端全面重构），但 task_state.json 报告 `output_files_produced: []` 和 `failed`，形成误导。

## 根因分析

非单一 bug，而是三层面的系统缺陷叠加：

### 层面 1 — 验证策略一刀切
`_get_verification_steps()` 从 profile.yml 全局加载 `mvn -DskipTests compile`，对**所有任务**执行。前端 Vue 任务 `task-frontend-checkdialog-table` 也被迫跑 Maven 验证，一个 JAVA_HOME 问题同时炸掉所有并行任务。

### 层面 2 — 任务状态二值化
goose 成功写入代码 → 产出文件存在 → 编译验证失败 → 整任务标记 `failed`。
`output_files_produced` 字段被设为 `[]`（因为 verify 分支绕过了赋值）。
"代码已产出但编译环境有问题"和"代码都没写出来"被混为一谈。

### 层面 3 — 失败传播无阻断
Phase 4 任务失败后，pipeline.py 的顺序循环无条件启动 Phase 5。
Goose 在 05-build.yaml 中因同因（JAVA_HOME）进入编译失败→重试死循环，耗费 4m36s。

### D1 — JAVA_HOME 注入加固

**问题**：`shell=True` + `env=merged_env` dict + `set "JAVA_HOME=..."` 命令前缀三重注入，在某些 Windows 环境下仍失效（可能是 cmd /c 下 env dict 与 set 的交互问题，或 PATH 中缺少 `%JAVA_HOME%\bin`）。

**方案**（`_run_verification_step()` 内）：
```python
# 1. Pre-flight：验证 JAVA_HOME 是否有效
def _verify_java_home(java_home: str) -> bool:
    java_exe = os.path.join(java_home, "bin", "java.exe")
    try:
        result = subprocess.run([java_exe, "-version"], capture_output=True, timeout=10)
        return result.returncode == 0
    except Exception:
        return False

# 2. PATH 注入：让 Maven 能找到 javac
java_home_prefix = (
    f'set "JAVA_HOME={jh}" && '
    f'set "PATH={jh}\\bin;%PATH%" && '
)

# 3. 去掉 env dict（shell=True 下与 set 冲突），仅用命令前缀注入
proc = subprocess.run(wrapped_cmd, shell=True, capture_output=True, ...)
# 不传 env=merged_env
```

### D2 — 按任务类型选择验证策略

**问题**：`TaskConfig` 无验证策略字段，所有任务共享 `profile.commands`。

**方案**：
- `TaskConfig` 新增 `verification: str = "auto"` 字段
- `tasks.yaml` 支持显式配置
- 映射表：

| verification | category 推断 | 执行 |
|-------------|--------------|------|
| `"auto"` | backend-api → backend | compileOnly + test |
| `"auto"` | frontend → frontend | frontendCheck（npm run build）或跳过 |
| `"auto"` | test → none | 跳过 |
| `"backend"` | — | compileOnly + test |
| `"frontend"` | — | 跳过或 npm run build |
| `"none"` | — | 仅检查产出文件 |

### D3 — 三元任务状态

**问题**：completed（全部通过）和 failed（任何失败）二元划分丢信息。

**方案**（`task_state.py` 新增常量）：
```python
STATUS_COMPLETED = "completed"        # goose成功 + 产出存在 + 验证通过
STATUS_CODE_PRODUCED = "code_produced"  # goose成功 + 产出存在 + 验证失败
STATUS_FAILED_NO_OUTPUT = "failed_no_output"  # goose失败 或 产出缺失
```

`task_state.json` 每条记录新增字段：
```json
{
  "verification_result": {"compile": "failed", "test": "skipped"},
  "files_exist": {"RepairNoticeService.java": true},
  "code_diff_summary": "+40 lines in RepairNoticeService.java"
}
```

### D4 — Phase 5 前置守卫

**问题**：pipeline.py 顺序循环无阶段间前置检查。

**方案 A**（最小改动）— `pipeline.yaml` 加 `prerequisite` 字段：
```yaml
- id: phase5
  name: "编译与测试"
  prerequisite: phase4_tasks_completed
  ...
```

**方案 B**（推荐）— pipeline.py 进入 Phase 5 前检查 task_state.json：
```python
if stage.id == "phase5":
    task_state = read_task_state(ai_dev_dir)
    code_tasks = [t for t in task_state.values() 
                  if t["status"] in ("completed", "code_produced")]
    if not code_tasks:
        logger.warning("No tasks produced code, skipping Phase 5")
        continue
```

### D5 — Goose 循环检测

**问题**：心跳仅检测输出频率（60s 无输出告警），不分析内容模式。Goose 在编译失败→重试循环中反复输出相同错误，心跳无法识别。

**方案**（`executor.py` `_heartbeat_loop()` 内）：
```python
from collections import deque
recent_stderr = deque(maxlen=10)

# 检测连续重复模式
if len(recent_stderr) >= 5 and len(set(recent_stderr)) == 1:
    logger.critical("Goose appears stuck in a retry loop (same error repeated 5x)")
    proc.kill()
```

### D6 — 错误信息结构化

**问题**：`mark_failed(tid, str)` 丢失关键上下文。

**方案**：失败时写入结构化信息：
```python
state_mgr.mark_failed(tid, json.dumps({
    "phase": "verification" if verify_failed else "execution",
    "goose_exit_code": result.returncode,
    "verification_error": f"Verification failed: {label}",
    "command": last_command,
    "env_java_home": verify_env.get("JAVA_HOME", "not set"),
    "files_declared": list(task.output_files),
    "files_on_disk": {f: (sandbox_path / f).exists() for f in task.output_files},
}, ensure_ascii=False))
```
